# Design: Phase-Containment Escape-Rate Ledger

## Summary

Umbrella issue #761 asks a measurement question the adversarial review pipeline could not previously answer: for each review stage (design-challenge, plan-stress-test, code-review), what fraction of sustained findings escaped from an earlier phase that could have caught them, and what fraction were genuinely irreducible — catchable only at that stage? The maintainer's stated position is that no review is relaxed on a cost argument alone; a stage earns relaxation only when data shows its irreducible-catch rate has trended to ~0 over a rolling window with enough samples to trust.

The phase-containment ledger is the instrument that produces that data. Each sustained adversarial finding is annotated with `introduced_phase` (where the flaw originated), `catchable_phase` (the earliest phase it could reasonably have been caught), and `caught_stage` (which review actually caught it). The system was built across four issues:

- **#762** (sub-1) shipped the per-finding annotation schema, a hand-rolled parser/validator, a rolling-history walker with rollup, and a maintainer-facing report CLI.
- **#763** (sub-2) shipped the first escape-driven upstream fix: a four-quadrant grounding discipline in `design-exploration` that catches the class of design-phase miss the #760 seed entries recorded.
- **#772** shipped, closing all six findings PR #770's review deferred against the sub-1 reader: a capped (K=5) markerless-comment-page hunt with unbounded post-find pagination resume (772-D2), REST-fallback `createdAt` threading (restoring latest-annotation-wins dedup under REST), `finding_key` format validation (schema `pattern` + validator Rule 12, case-sensitive), total `Truncated`/`InvalidEntryCount` telemetry across every return shape of both public fetch functions (772-D1), and pair-matched open/close marker-block parsing so an unclosed block no longer corrupts the following block (772-D3). A post-implementation code-review pass (commit `6626f2f`) sustained four more findings — M1/M2 closed two additional silent-truncation paths (GraphQL post-marker-find pagination failures, and a REST list-level `gh` call failure that previously had no `else` branch at all), M4 folded D6's parser-level block skips into `InvalidEntryCount`, and M8 replaced a culture-hardcoded test literal. One deferred-significant maintainability finding (Surface A/B marker-hunt logic duplication) was filed as issue #831, not fixed in #772.
- **#782** closed the gap between "emission is specified" and "emission happens" for the design-challenge and code-review surfaces: a warn-only sweep that detects sustained findings missing their paired ledger blocks, plus the first organic backfill (16 entries recovered from PRs #775/#778/#781). It left the plan-stress-test surface's emission check silently false-clean (`sustained=0 blocks=0` on every plan, compliant or not) because it was written against a `judge-rulings` machine-block contract the plan-authoring writer never actually emitted.
- **#811** (this branch) fixed the plan-stress-test surface's false-clean: Issue-Planner now emits a machine-readable `judge-rulings` block alongside the human prose, the reader gains an honest fallback and fail-loud-on-duplicate-heads behavior, and issue #794 was backfilled as the acceptance-criteria witness (`sustained=11 blocks=11`, verified live).
- **#768** added the ledger's review-**cost** half, alongside its existing review-**value** half: the maintainer report now also shows, per stage, a signal-to-noise line — design-challenge dismiss-rate over dispositioned findings; code-review's defense-kill rate and post-judge dismiss-rate (two distinctly labeled noise columns, never conflated) plus a defer count; plan-stress-test's defense-kill rate (data accrues from #811 forward) — and, within the code-review stage, a per-reviewer-source kept-vs-dismissed table keyed on a new schema v3 `reviewer_source` field (absent groups as `local`). This is the measurement instrument #830's Gate A→B needs to judge whether an outside PR reviewer earns its seat, presented next to the existing escape/irreducible-catch rates without changing their meaning or the relaxation-eligibility formula (presentation-only). An adversarial code review of the implementation (5-pass panel → defense → judge) sustained 20 of 22 findings; six were judged must-fix and closed in a post-fix pass (commit `702b185`) that also hardened the subsystem's hand-rolled parsers against a decoy class the review found: YAML block-scalar (`key: |`/`key: >`) string content — e.g. a `disposition_rationale` value, or judge-rulings vocabulary, quoted inside a block scalar — was previously read as real structure, letting attacker-controlled comment text fabricate phantom disposition entries or defense-kill contributions (a marker-forgery vector). The remaining 14 lower-severity findings were deferred as issue #842; one (remote-keyed re-review dedup semantics) was routed to the maintainer as a plan-conformance question rather than treated as a code defect (see Known Limitations below). Full suite: 3579 passed, 0 failed, 28 skipped.
- **#834** (Codex adoption A1) wired reviewer attribution end-to-end, fulfilling the 768-D8 deferred coupling: `skills/code-review-intake/SKILL.md` now records `reviewer_source` per finding at ingest time, normalized per the canonical rule stated there (see `skills/code-review-intake/SKILL.md` § GitHub Review Mode — this doc does not restate the mechanism). `skills/review-judgment/SKILL.md` gained a per-finding `stable_finding_key` discriminator for multi-finding GitHub comments (`gh-{comment_id}-{hash}` alongside the unchanged single-finding `gh-{comment_id}`) and a 3-tier `reviewer_source` lookup order (in-context intake ledger → `gh api` re-derivation → `unresolved` sentinel), with the writer prose now actually populating `reviewer_source` from each finding's origin instead of hardcoding `local`.

This document describes the current shipped system. It does not cover relaxation-criterion governance (deferred to the umbrella, gated on this data).

## Design Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | Annotation home | One uniform `<!-- phase-containment-{ID} -->` block per finding, written by whichever stage caught it onto the surface it already posts to | The #762 design challenge found the plan-stress-test surface has no parseable block and `judge-rulings` is parsed by a bespoke line-scanner, not YAML — in-place extension of `finding_dispositions`/`judge-rulings` could not hold uniformly across all three surfaces |
| D2 | Escape-distance arithmetic | `escape_distance = projection(caught_stage) − ordinal(catchable_phase)`, validated by recomputation and rejected (not clamped) if the stored value differs, or if `introduced_phase > catchable_phase`, or if `catchable_phase` exceeds the caught stage's projection | A silently-clamped or unchecked distance would corrupt the escape/irreducible split the whole metric depends on |
| D3 | Fail-loud over silent-zero | Any unparseable, ambiguous, or unknown-vocabulary content under a recognized marker head is `could-not-verify`, never silently treated as zero findings or a clean gap | A metric whose failure mode is silent zero would misreport a stage as clean when it was actually unmeasured — the exact false-clean scenario S2 in #772/#782 exists to prevent |
| D4 | Undersample guard granularity | Per-stage `insufficient_data` (n<5) withholds that stage's relaxation-eligibility signal independently — not a single global window-count guard | A global guard would let a near-empty stage (e.g. plan-stress-test at n=1) read as computable simply because other stages had volume |
| D5 | `finding_key` surface discrimination | Uniform `{surface}:{stable_finding_key}` format; `Get-EmissionGap` requires a block's `finding_key` to be prefixed for the surface being checked before counting it toward that surface's block count | design-challenge and plan-stress-test marker heads can legitimately co-occur on the same issue body; without the prefix check, a block emitted for one surface could silently satisfy another surface's gap count |
| D6 | Warn-only, never-blocking posture | The emission-check sweep only ever posts an advisory comment; enforce mode is defined in the `-Mode` parameter but explicitly unimplemented (`warn` behavior runs regardless of the value supplied) | Mirrors the frame-credit-ledger warn→enforce staging precedent — surface the gap, never gate a review, PR, or issue update on it |
| D7 | Why emission enforcement needed a separate sweep | #762 shipped emission as a prose post-step in three skills (`design-exploration`, `plan-authoring`, `review-judgment`); every real review after PR #770 merged — #775, #778, #781, sustaining 13+ findings between them — emitted zero paired `phase-containment` blocks | An unenforced terminal instruction is reliably skipped; #782 built a reader-side reconciliation sweep as the durable guarantee rather than trusting a stronger prose reminder to succeed where the first one failed |
| D8 | Marker-vocabulary gating (not bare substring match) | A comment only counts as an authoritative judge-rulings/finding_dispositions surface when its marker head is followed, within a bounded lookahead window, by recognizable field-vocabulary tokens (`disposition`, `judge_ruling`, `verdict`, `finding_key`, `finding_id`, `schema_version`) at a real YAML key position (line-start, flow-mapping, or dash-space list-item) | Early iterations mistook ordinary prose describing the marker convention, or dash-space list-item findings, for real marker content — both a false-could-not-verify and a false-clean failure mode had to be closed |
| D9 | Backfill-append primitive is read-modify-write, never upsert | `Add-CommentBlocks` does `gh api` GET → verify marker → concatenate → PATCH → positive-proof re-verify; it explicitly does not use `Find-OrUpsertComment`, whose PATCH path replaces the body verbatim | The judge-rulings YAML that Code-Conductor's credit harvest reads lives in the same comment the backfill blocks are appended to — an upsert-style replace would destroy it |
| D10 | Post-write verification is positive-proof, not byte-prefix | Verification confirms (a) the original marker survived and (b) every appended block is present, parseable, and content-identical in the re-fetched body — not an exact `StartsWith` byte comparison | Live backfill against #775/#778/#781 showed GitHub's API benignly normalizes whitespace on write/read, which broke ordinal-prefix comparison and produced false negatives that would have trained the caller to ignore a real fail-loud signal |
| 811-D1 | Plan-stress-test writer emits a machine `judge-rulings` block, not a prose parser | `skills/plan-authoring/SKILL.md` § Judge-rulings machine block: at Post-Judge Reconciliation, Issue-Planner appends a `<!-- judge-rulings` block (one entry per merged `finding_id`, binary `sustained \| defense-sustained` projection, atomic single write) at the end of the `<!-- plan-issue-{ID} -->` comment, alongside the unchanged human-facing `**Plan Stress-Test**` prose | #782's design already specified that the plan-stress-test surface persists a `judge-rulings` block and the reader's parser branch was written against that contract — the actual defect was a writer deviation (the block was never emitted), not a missing parser. Live #794 prose also proved not machine-countable (aggregated bullets, out-of-enum vocabulary), which foreclosed a prose-parser fix |
| 811-D2 | Reader honest fallback, differentiated by cause | `Test-EmissionMarkerPresent`, plan-stress-test surface only: a body carrying `<!-- plan-issue-` and a line-start `**Plan Stress-Test**` heading but no judge-rulings head reports could-not-verify instead of falling through to `sustained=0` chatter-skip. `Format-EmissionGapLine` renders two distinct could-not-verify messages — `head-missing` (blocks may already exist; a one-line backfill candidate) vs. `head-corrupt` (a machine head exists but failed to parse) — and renders a negative gap (more blocks than sustained) as a loud excess, never as `clean` | A prose-only plan (every plan persisted before #811) previously read as ordinary chatter and silently contributed `0/0`, indistinguishable from an issue with no stress-test history at all — exactly the false-clean scenario S2/S3 this issue was filed to close |
| 811-D3 | Fail-loud on duplicate judge-rulings heads (not latest-wins) | When two or more real (vocab-gated) judge-rulings heads exist in one comment body, the reader returns `could-not-verify` rather than picking one | Owner decision during the plan's own adversarial stress-test: latest-wins fights the existing ambiguity-walk / attributed-vs-bare head precedence logic, and the writer's replace-own-block rule already guarantees a single head on the normal persist path — a duplicate signals a genuine anomaly (e.g. a double-run backfill) that should surface loud, not resolve quietly |
| 811-D4 | Historic corpus: no bulk backfill, #794 only | Only issue #794 was backfilled (17 entries via the sibling `Add-JudgeRulingsBlock` primitive) as the acceptance-criteria witness; the other ~34 of ~39 closed plan issues stay prose-only and render the differentiated could-not-verify variant until they age out of the 90-day corpus window | Honest backfill of historic plans requires per-issue prose re-reading; a mechanical block-derived bulk backfill would be tautologically clean (sustained ≡ blocks by construction) and would mask real historic under-emission — owner decision `historic-corpus-disposition-811` |
| 772-D1 | Truncation/invalid-entry telemetry needed a real return channel | Reshaped the private fetch helpers (`Get-SurfaceACorpusGraphQL`/`Get-SurfaceBCorpusGraphQL`/`Get-PhaseContainmentCorpusRest`) and `Invoke-PhaseContainmentCommentScan` from bare arrays/`$null` into carrier objects (`{Tuples/Entries; Truncated; InvalidEntryCount; IsError}`, explicit `IsError` replacing the old `$null`-means-REST-fallback sentinel), then threaded `Truncated`/`InvalidEntryCount` through all 8 `Get-PhaseContainmentHistory` returns and all 7 `Get-PhaseContainmentCommentCorpus` returns | A bare-array/`$null` return had no channel to carry a truncation signal out of the private fetch layer; the plan's own stress-test flagged this as the load-bearing propagation gap (challenge P2, sustained) that every other truncation fix in the plan depended on |
| 772-D2 | Marker hunt is capped (K=5), not unbounded — but post-find collection is not | `Get-SurfaceACorpusGraphQL`/`Get-SurfaceBCorpusGraphQL` hunt up to 5 additional comment pages for a phase marker before giving up (GraphQL only; REST returns all comments in one call); once a marker is found, pagination resumes UNBOUNDED to collect all remaining comment pages before recording the tuple | An unbounded hunt on every markerless thread would regress the primary read path's performance (F11's no-perf-regression clause); the plan-stress-test caught that capping the hunt without also uncapping post-find collection would silently drop a `phase-containment-{N}` block sitting past page 5 on a thread where the marker itself was found early (challenge P6, sustained) |
| 772-D3 | Parser-level block skip must warn, not just drop | `Get-PhaseContainmentBlock` pair-matches open/close marker tags (an unmatched open tag followed by a later open tag drops the earlier block as malformed) and now emits `Write-Warning` plus an optional `-SkippedCount` `[ref]` out-param that `Invoke-PhaseContainmentCommentScan` folds into `InvalidEntryCount` | The design-challenge caught that D6's original fix — silently dropping the unclosed block — was itself a new silent-degradation path if left unflagged (challenge P10, sustained), inconsistent with the ledger's own "no path lacks a fix, flag, or fail-closed outcome" bar |
| 768-D1 | Cost data source | Rides the existing `Get-PhaseContainmentCommentCorpus` tuples (rolling-history core); no new fetch machinery | A second fetch stack would duplicate the marker-hunt machinery #831 already exists to deduplicate |
| 768-D2 | Placement (two-file cut) and joint routing | Parsing lands additively in `phase-containment-emission-check-core.ps1` (new public `Get-DispositionTally`, reusing the file's `script:`-scoped block-scalar/head-detection internals via additive-return extension); aggregation and rendering land in new `phase-containment-cost-core.ps1`; `phase-containment-rolling-history-core.ps1` stays frozen at ≈0-line diff. `Get-ReviewCostRollup` classifies each body by its own `(tuple Surface, marker head)` pair — never head alone (judge-sustained M1) — because code-review and plan-stress-test judge-rulings share the byte-identical `<!-- judge-rulings` head; the parsed result is then routed to the correct stage bucket by this function | The parse internals are file-private (`script:`-scoped); duplicating them in the cost core would fork the subsystem's hardened parsing and recreate the #831 duplication disease. Routing by head alone, rather than the joint (Surface, head) pair, would misattribute a plan-stress-test judge-rulings body's counts to code-review or vice versa |
| 768-D3 | Parsing discipline + field naming | Hand-rolled line-regex parsing only — no `powershell-yaml`/`ConvertFrom-Yaml` anywhere in the subsystem; schema v3's new attribution field is named `reviewer_source`, not `source` | Standing security prohibition (billion-laughs DoS on untrusted comment bodies); a flat `source:` key scan under the mandated parser would collide with the existing nested `ac_cross_check.source` enum, misreading AC cross-check evidence values (`issue`, `pr-body`) as reviewer identities |
| 768-D4 | Two-labeled-noise-column semantics | Two noise metrics, never conflated under one label: **defense-kill rate** (`defense-sustained / (sustained + defense-sustained)`, from judge-rulings; rendered for code-review AND plan-stress-test) and **post-judge dismiss-rate** (`dismiss / dispositioned`, from review-dispositions; code-review only). `defer` counts in the denominator, not the noise numerator, as its own column (code-review only, the only surface whose disposition vocabulary has `defer`) | Defense-killed findings never reach review-dispositions but are durably recorded in judge-rulings; measuring them for one stage and not the other would compare unlike quantities. Counting `defer` as noise would overstate noise for exactly the findings a loud guard flagged as possibly real |
| 768-D5 | Corpus symmetry + forward gap | Cost is computed over the identical judge-rulings-discovered PR population as the value ledger; PRs with value data but no review-dispositions marker render as a per-stage count line ("PRs with value data but no cost marker: N") | Same-population comparison is what the relaxation decision needs; the forward gap is rendered, not hidden |
| 768-D6 | Fetch-coherence default-bypass | Cost-enabled runs (the default) bypass the value path's 1-hour cache so both fetches are same-run fresh; the cache-bypass edit happens before the value fetch call, not appended after render. `-ValueCacheOk` restores cached mode and prints an explicit population-mismatch caveat | A warm cache would let the two report halves silently describe different PR sets — exactly the comparison-invalidation 768-D5's corpus-symmetry guarantee exists to prevent |
| 768-D7 | Fail-loud + isolation | Corpus fetch failures render "COST DATA UNAVAILABLE (fetch {source})" — never INSUFFICIENT DATA, keeping "fetch failed" distinct from "too few findings"; re-review rounds dedup latest-wins (review-dispositions: per-PR per-`stable_finding_key` merge; judge-rulings/finding_dispositions: per-PR/issue latest marker generation via `Select-LatestByCreatedAt`), mirroring `Invoke-PhaseContainmentDedup`'s convention; the entire cost path is wrapped in try/catch so any failure degrades to "cost section unavailable: {reason}" with the value ledger always rendering first | Ledger D3 (fail-loud over silent zero) and the trusted value surface must survive any cost-path defect |
| 768-D8 | #834 coupling kept generic | Schema v3's `reviewer_source` accepts any string; the review-judgment writer prose now populates it from the finding's actual origin (shipped by #834): `local` for pipeline-native findings, or a normalized external identity resolved through a 3-tier lookup order (in-context intake ledger → `gh api` re-derivation → `unresolved` sentinel on failure) for GitHub-sourced findings | #834 owns the intake tagging convention for external-reviewer identity naming; wiring its wording here ahead of shipping would have drifted if the convention changed before it shipped — this deferral is now resolved |
| 817-D1 | Diagnostic containment at the aggregation seam, not a gate rewrite | New private `Get-JudgeRulingsDuplicateDiagnosis` diagnoses WHY the M1 duplicate-head guard (`Get-JudgeRulingsIsolatedRegion`) fired: it re-runs the vocab gate per candidate with each non-last candidate's window additionally truncated at the next candidate's start — 1 survivor means `window-bleed` (one real block, seen twice), ≥2 survivors means `genuine-duplicate`. Called only from `Get-EmissionGap`'s per-body could-not-verify classification; `Get-SustainedFindingCount`'s pinned `{ParseStatus, SustainedCount}` projection (AC8 regression pin) is untouched | The M1 guard's fixed 400-char lookahead window (issue #817's root cause) cannot itself tell a genuine second head from a decoy whose window bled into a real block's own vocabulary; bubbling a fix through the sustained-count return chain would break the AC8-pinned output shape and burden 10+ StrictMode could-not-verify return sites, so an aggregation-seam helper was used instead |
| 817-D2 | Per-body decoy-aware reason derivation | `Get-EmissionGap`'s per-body classification: a could-not-verify body with ≥2 vocab-gated heads whose diagnosis is `window-bleed` contributes `decoy-ambiguous`, never `head-corrupt` — the pre-existing blind `$hasRealHead → $sawRealHeadCorrupt` path would otherwise shadow the honest label. Genuine-duplicate bodies keep `head-corrupt`. The cross-body aggregation priority `head-corrupt > decoy-ambiguous > head-missing` applies only across bodies, never within one | A maintainer reading `head-corrupt` reasonably investigates content corruption that does not exist; the near-decoy case is a text-layout quirk, not a parse failure, and conflating the two costs a debugging session for nothing |
| 817-D3 | Reason enum extension + surface-scoped render | `Reason` enum becomes `ok \| head-missing \| head-corrupt \| decoy-ambiguous`; `Format-EmissionGapLine` gains a hedged plan-stress-test-only render line naming nearby-mention ambiguity, not content corruption. Code-review/design-challenge render wording stays byte-for-byte unchanged | Extending the differentiated render to every surface would violate the settled 811-D1 byte-for-byte invariant; Reason stays field-readable there instead |
| 817-D4 | One accepted residual (T6) + one confirmed-correct edge case (T7) | (1) **T6 — close genuine duplicates**: two real heads separated by little or no text, where the first head's own gate-vocabulary sits textually past the second head's start; truncation empties the first candidate's window before it reaches its own vocabulary, so only the second (last, untruncated) candidate survives, and the pair is labeled `decoy-ambiguous` instead of the correct `head-corrupt` — a genuine diagnostic limitation, accepted and regression-pinned, not eliminated. (2) **T7 — mirror ordering**: a decoy embedded after a real head but before that head's own trailing vocabulary; the real head's window truncates at the decoy's start (0 survivors from that candidate) while the decoy, as the last candidate, keeps its full window and survives, so the pair is also labeled `decoy-ambiguous` — but this IS the objectively correct diagnosis for a single real block plus a nearby decoy, exactly the near-decoy shape the mechanism exists to catch. T7 is not a residual; it confirms the mechanism generalizes correctly to a decoy placed after, not just before, the real head. (The design record originally mischaracterized this placement as `head-corrupt`; a mechanical trace during test authoring corrected it to `decoy-ambiguous`, with no implementation change required.) | T6 is a rare, real-world-unlikely layout; the lite scope (owner decision `lite-scope-bias-817`) accepts documenting and fixture-pinning it over the deeper parsing that would be needed to disambiguate close-adjacent genuine duplicates. T7 needed no such acceptance — it is regression-pinned as a correctness proof, not a scope tradeoff |

## Issue #863: Plan-Comment-Cap Split — Priced Decision Record

This section is the full priced-decision record for issue #863 (umbrella #761). It exists so a future reader never has to re-open #863, re-run its design challenge, or re-litigate any of the following.

### The cap and its empirical provenance

GitHub's comment-body size limit is **65,536 codepoints — not bytes**. This is not a documented GitHub API contract; it was confirmed empirically. Issue #854's `plan-issue-854` comment measured **65,377 codepoints and 65,581 UTF-8 bytes** and existed live on GitHub. A 65,581-byte body existing on the platform proves the cap cannot be byte-denominated (a byte cap at or below 65,536 would have rejected that write); the only cap consistent with the observed acceptance is a 65,536-codepoint cap. The only size contract that existed in the tree before this issue — the "if the payload size approaches 60,000 bytes, you MUST emit a warning" prose in `agents/Issue-Planner.agent.md`, `agents/Solution-Designer.agent.md`, and `agents/Experience-Owner.agent.md` — is denominated in the wrong unit for this reason. Correcting that prose is out of scope for #863 and travels with the deferred overflow-paging work (863-D9 in the plan; tracked separately), since that is where the threshold itself gets redesigned. The literal value `65,536` did not appear anywhere in the tree before this issue; this document is what puts it there (AC1).

### #854's measurement (the motivating incident)

Issue #854's plan comment landed at 65,377 of 65,536 codepoints — **159 codepoints of headroom** — after the authoring agent spent 5–10 rounds compressing load-bearing plan content to fit under the cap. The plan that came out of that compression was worse than the plan that went in, and nothing in the pipeline recorded that degradation; it was only surfaced by an owner question during the PR #857 review session, then measured.

Per-family composition of that comment:

| Family | Codepoints | Share | Reader | Lifecycle |
| --- | --- | --- | --- | --- |
| Plan prose | ~27,362 | 41.9% | Conductor, human — whole-read | written once |
| `frame-slice` ×11 | 23,074 | 35.3% | Specialists — one at a time, script-mediated | written once |
| `phase-containment` ×36 | 10,776 | 16.5% | Measurement scripts only — no agent reader | written after the plan |
| `frame-spine` | 2,204 | 3.4% | Conductor, Spine-Runner — whole-read | written once |
| `judge-rulings` (plan-surface) | 1,961 | 3.0% | Measurement scripts only — no agent reader | written after the plan |

Non-prose payload totals **58%** of the comment. Plan content alone (prose + spine, the only families a human or Conductor ever reads in full) is ≈29,566 codepoints — comfortably under the cap with ~35,970 codepoints of headroom. It is the non-prose, script-only-read families (`frame-slice`, `phase-containment`, `judge-rulings`) that pushed #854 into the wall.

### The retired 5.6x figure and why

Issue #863 was originally filed asking a co-location-tax question: framed as measurement-readers fetching roughly 5.6x more comment content than they structurally need, because of family co-location inside one comment. **This framing was retired during the design phase and re-scoped as unactionable** (owner decision `d-scope-capacity`, recorded on #863's issue body), not carried forward into the shipped design. The reason: both measurement readers that would have benefited from a leaner fetch —

- `phase-containment-rolling-history-core.ps1`'s GraphQL/REST corpus fetch (`Get-PhaseContainmentCommentCorpus`, `Get-PhaseContainmentHistory`), and
- `phase-containment-emission-check.ps1 -Issue N` / `-Pr N` (via `Get-EmissionGap`)

both fetch **every comment on the issue** as their unit of work — they have no API-level mechanism to sub-select within a single comment body, and splitting content across sibling comments does not change that: fetching N comments instead of 1 comment is not fewer bytes over the wire, and neither reader was ever byte-metered in a way the split could improve. The 5.6x figure priced a fetch that no reader in the tree actually performs. Splitting the plan comment by reader was retained — and shipped — for a different, real reason: relieving write-side pressure on the 65,536-codepoint cap (the #854 incident), not for a read-side cost reduction that was never real.

### The durability trade-off

Before #863, every annotation family lived in one comment (`<!-- plan-issue-{ID} -->`) on the subject issue. That co-location bought two properties that #863's split had to preserve or the change would not have been worth making:

1. **No drift, no second store.** Every annotation still lives as a comment on its subject issue — there is no separate database, file, or external service to keep in sync with the issue, and no risk of the annotation and the issue silently diverging (e.g., issue deleted/transferred while an external record survives, or vice versa). The split adds sibling *comments*, which are still comments on the same issue; it does not introduce a second store.
2. **Fix A's forgery gate (issue #782) stays intact.** `phase-containment-emission-check-core.ps1`'s co-location gate (condition 1, `:2404-2420`) requires that a `phase-containment` block only counts toward a surface's sustained-finding tally when the *same comment body* also carries that surface's authoritative `judge-rulings` (or `finding_dispositions`) head. This closes a forgery vector where an attacker (or a stale re-sweep) could inject phase-containment-shaped blocks onto a comment that never actually carried a real judged head, artificially inflating or deflating a stage's measured count. The gate is keyed on **co-location within one comment body**, not on comment authorship.

The chosen split shape — siblings on the **same issue**, with `judge-rulings` co-moving to the same sibling as the `phase-containment` blocks it authorizes (863-D4) — preserves both properties unchanged: property 1 holds because siblings are still issue comments on the subject issue, not a second store; property 2 holds because the sibling's own body still carries its own authoritative head, so Fix A's condition 1 evaluates exactly as it did pre-split, just against the sibling's body instead of the plan comment's body. Neither the schema nor the code of the co-location gate was touched by #863.

### The 811-D1 suppression gate's forgery model (M2 fix, judge-sustained code-review)

`Get-EmissionGap`'s 811-D1 fallback (`$anySiblingHasRealHead` in `phase-containment-emission-check-core.ps1`) suppresses a plan body's would-be `head-missing` contribution when the plan's ledger legitimately moved to a sibling comment. The population this gate must reject: a **stale or foreign real judge-rulings head** — a genuine, vocab-gate-passing head left on the issue by an earlier plan run, a different plan entirely, or any other real-head-bearing surface not bound to the current plan's designated ledger sibling. A pre-fix version of this gate counted ANY real head found anywhere in the fetched `-Bodies` set as sufficient to suppress, which meant a stale prior-generation head could falsely suppress `head-missing` even when the CURRENT plan's actual ledger sibling was deleted, corrupt, or never written — a false-clean the same class 863-D11 (writer-emitted pointer, not absence-inferred) was built to prevent, reachable via a different door than the one 863-D11 itself covers. The fix binds the suppression gate to the plan's own `<!-- phase-containment-ledger-{Id} -->` identity marker (the `-Id` already available in `Get-EmissionGap`'s scope): a body only qualifies as the plan's genuine sibling when it carries BOTH a real judge-rulings head AND that marker for the current plan's own `Id`. A real head belonging to a foreign plan run is not evidence that THIS plan's sibling exists, so it can no longer suppress. This closes the vector while keeping the check at the aggregation seam (never inside `Test-EmissionMarkerPresent`, whose single-`[string]$Body` parameter cannot ask "did some other qualifying body supply the real head").

### What was rejected (for future non-relitigation)

Three alternative designs were considered and rejected during #863's plan phase; none should be re-proposed without new evidence, per each decision's stated shift trigger:

- **The two-body core-lookup change.** Teaching `frame-spine-core.ps1` and/or `frame-validate-core.ps1` to natively read and merge two comment bodies, rather than having callers concatenate client-side. Rejected as `e1-da-transport-revised` once plan-phase discovery found `frame-validate-core.ps1` has its own **independent** duplicate parser and regex (separate from `frame-spine-core.ps1`'s), meaning a two-body core change would need a second co-parsing site, plus three levels of non-composing parameter sets (`ByPath`/`ByStdin` combinations) across the two files. The benefit being purchased — fewer reader bytes fetched — was already shown to be non-real by the retired 5.6x analysis above, so the price was rejected in favor of the cheaper concatenation-based design (863-D2).
- **Re-basing Fix A's forgery gate onto authorship instead of co-location.** Considered and rejected as 863-D5. This was judged orthogonal to what the co-location split actually needed to preserve, and would have been a net security regression: an authorship-based gate trusts whoever authored the comment rather than requiring the surface's own authoritative head to be textually present in the same body, which is a weaker guarantee than what Fix A currently provides.
- **A spine copy inside the sibling comment.** Duplicating the `frame-spine` block (or a subset of it) into the `frame-slices-{ID}` sibling, so a sibling-only reader would not need to also fetch the plan comment. Rejected because it would have created exactly the second store 863-D2's design explicitly avoids — the spine and the slices it routes to would then need to be kept in sync across two independently-editable comment bodies, reopening the drift risk the durability trade-off above requires closing.

### AC11 — the `frame-validate -Mode plan` manual-tool note

`frame-validate -Mode plan` is a **manual-only** tool: nothing in CI or any automated path invokes it against a comment body. `quick-validate-core.ps1` calls `Invoke-FrameValidate -RootPath $RootPath` with no `-CommentBody` argument, which runs `frame-validate` in its default (non-plan) mode; no automated caller ever supplies a plan comment body to `-Mode plan`.

If a maintainer manually runs `frame-validate -Mode plan` against a **post-split plan comment body without first concatenating the `frame-slices-{ID}` sibling**, the tool produces a real structural violation — not a false positive, but a confusing one if the maintainer does not already know about the split. `Get-FVPlanSliceBlock -CommentBody $commentBody` finds zero `<!-- frame-slice ... -->` blocks (they moved to the sibling), so every port the spine references comes up with no matching slice, and `frame-validate-core.ps1:482` emits, once per affected port/step pair, the following literal (quoted verbatim from the source — `$portName` and `$stepId` are PowerShell string-interpolation placeholders, not literal text):

```text
port $portName references step $stepId, but no frame-slice provides: anchor exists; add slice provides: [$portName].
```

For example, against issue #863's own spine, a bare (unconcatenated) plan-body run would render this as `port release-hygiene references step s8, but no frame-slice provides: anchor exists; add slice provides: [release-hygiene].` (and similarly for every other port). A prior draft of this exact doc note was found to have paraphrased this string incorrectly — it dropped the colon after `provides` and omitted the entire remedy clause (`add slice provides: [...]`) — which meant a maintainer grepping the doc note against real tool output would find no match. That paraphrase error is the precise confusion AC11 exists to prevent, which is why this note quotes the source line directly rather than restating it.

This is expected, not a defect: the maintainer needs to concatenate the `frame-slices-{ID}` sibling onto the plan comment body first — the same two-fetch-and-concatenate pattern the `frame-spine-lookup` shim already uses for its automated callers — before invoking `frame-validate -Mode plan`. This doc note is what a maintainer troubleshooting that exact error message would search for.

### Cross-version transition window

Live agents resolve `skills/plan-authoring/SKILL.md` and `agents/Issue-Planner.agent.md` from the **installed plugin cache** (see `CLAUDE.md` § Releases), not from the source repo's working tree. This means the new writer shape (siblings, pointers, the `frame-slices-generated-at` stamp) only reaches a live authoring agent after a plugin version bump ships and the consuming installation runs `claude plugin update` — the code landing in this repo's `main` branch is necessary but not sufficient for the new shape to actually be written anywhere.

**Accepted failure mode, recorded as a decision, not left unexamined (M8 fix: two distinct skew directions, not one).** A reader meeting a **new-shape** plan comment (one whose `frame-spine` block carries a populated `slice_comment_id`) under version skew fails loud with one of two distinct statuses, depending on which side of the reader is stale:

- **`sibling-unstamped`** (the more common skew case): the core script (`frame-spine-core.ps1`) IS `#863`-aware, but the concatenated body it was handed never actually includes the sibling — e.g. a stale-cache dispatch context (Code-Conductor or a resumed `/implement` path running an unrefreshed plugin cache) that omits forwarding `slice_comment_id` to the specialist, so the specialist's `frame-spine-lookup` shim takes its "no sibling id was dispatched" single-fetch branch even though the plan itself is genuinely split. `Get-FSCSliceSiblingGeneratedAt` finds no `frame-slices-generated-at` stamp in the un-concatenated body and `Invoke-FSCSpineLookupCli` returns `status: sibling-unstamped` (exit code 1), naming the `slice_comment_id` it found but could not corroborate.
- **`missing-slice`** (the less common case): an **older cached reader** running the full pre-#863 `frame-spine-lookup` contract — core script included — has no sibling-drift-check code at all, so it never even attempts to detect the split; it single-fetches the plan comment body and calls the lookup directly. Since the `frame-slice` block for the requested step now lives only on the sibling, `Get-FSCSliceBlocksByStepId` finds zero matches and `Invoke-FSCSpineLookupCli` returns `status: missing-slice` (exit code 1), naming the missing step id.

Both are **loud, explicit failures**, never a silent wrong answer or a stale/partial dispatch. Both failure modes are accepted as-is for the cross-version window; no forced-upgrade mechanism, version-skew detector, or compatibility shim was built to prevent a stale reader (in either sense above) from meeting a new-shape plan. A maintainer who hits either `sibling-unstamped` or `missing-slice` against a plan comment they know is post-#863-shaped should run `claude plugin update` before re-dispatching.

## Metric Definitions and Scope Boundary

The ledger measures, per review stage, over a rolling window:

- **Escape rate** — the fraction of findings whose `catchable_phase` denominator stage leaked (caught later than that stage).
- **Irreducible-catch rate** — the fraction caught exactly at their `catchable_phase` stage, genuinely requiring that stage's information.
- **Relaxation-eligibility signal** — emitted only when irreducible rate trends to ~0 **and** the stage's window sample size is `n ≥ 5` **and** no critical- or high-severity finding was caught in the window (#854 M4 fixed the veto to check both severities; it previously checked `critical` only). Below n=5 the signal is withheld and the stage reports `INSUFFICIENT DATA` rather than a false confident rate.

The system is explicitly a measurement instrument, not an enforcement or relaxation mechanism:

- It never blocks a PR, issue, or review on missing or gapped ledger data (D6).
- It does not itself decide to relax any review; that decision is deferred to the umbrella once the data supports it.
- The emission-check sweep does not verify comment authorship — any comment matching the recognized marker shapes is trusted at face value; this is an accepted residual-risk tier for a warn-only maintainer advisory, not a compliance guarantee.

## Current Status and Known Limitations

Corpus totals have grown substantially since this document was first written and now reflect accumulated activity across many issues and PRs rather than the small seed+organic counts originally recorded here; see `phase-containment-report.ps1`'s live output for current per-stage `n`, escape rate, and irreducible rate. This section focuses on what #811 changed for the plan-stress-test surface specifically.

Before #811, the plan-stress-test surface's emission check (`phase-containment-emission-check.ps1`) was silently false-clean: it reported `sustained=0 blocks=0` on every plan regardless of how many sustained findings and phase-containment blocks actually existed, because it parsed the surface with the code-review `judge-rulings` YAML parser against a plan comment that only ever carried human prose bullets. Issue #794 was the live reproduction (11 sustained findings, 11 real blocks, reported as `0/0`) and is now the shipped fix's acceptance-criteria witness — after the plan-authoring writer change and the #811-D4 backfill, `phase-containment-emission-check.ps1 -Issue 794` reports `plan-stress-test #794: clean -- sustained=11 blocks=11` (verified live). The plan-stress-test surface's trend baseline (relaxation-eligibility signal, escape rate) starts accumulating organic, honestly-counted data from #811's ship date forward; historic prose-only plans are not retroactively counted (811-D4) and age out of the rolling window naturally.

Known limitations:

- **Surface A/B marker-hunt logic is duplicated between the two GraphQL fetch helpers.** The K=5 capped hunt + unbounded-resume-after-find state machine (772-D2) was added near-verbatim (~65-90 lines) to both `Get-SurfaceACorpusGraphQL` and `Get-SurfaceBCorpusGraphQL` in `phase-containment-rolling-history-core.ps1`, extending a pre-existing Surface A/B mirroring pattern already in the file; Surface B's Pester coverage of the five hunt dispositions (marker-on-page-1, marker-found-mid-hunt, cap-exhausted, mid-hunt-timeout, natural-exhaustion) is also thinner than Surface A's. The #772 post-fix code-review judge ruled this non-blocking for merge (defense-sustained, categorized structural) since no #772 acceptance criterion covered extraction; it was filed as issue #831 as a maintainability follow-up, not a correctness gap.
- **The emission-check's single-target mode has no dry-run flag.** `phase-containment-emission-check.ps1 -Pr N` or `-Issue N` always posts its advisory report via `Find-OrUpsertComment` — there is no read-only mode that computes the gap without writing a comment. This was discovered during #782's CE Gate pass. It is a usability gap worth fixing, not a defect: the report is idempotent (find-or-upsert on a fixed marker) and warn-only, so repeated runs do not compound, but a maintainer who wants to preview the gap without touching the PR/issue currently cannot.
- **Historic plan-stress-test corpus is prose-only and will read as could-not-verify.** ~34 of ~39 closed plan issues predate #811's writer change and carry no machine `judge-rulings` block. `Format-EmissionGapLine` renders these as the differentiated `head-missing` could-not-verify variant (naming the existing block count as a one-line backfill candidate) rather than a false `clean`, but no bulk backfill was performed (811-D4, owner decision) — only #794 was backfilled as the AC witness.
- **14 low-severity review-cost findings deferred to #842.** The #768 implementation's adversarial code review sustained 20 of 22 findings; the 6 judged must-fix are already closed (see the #768 summary bullet above). The other 14 — dedup edge cases for remote-keyed re-review rounds, drift-guard gaps that don't scan every duplicated copy, and display/documentation nits (e.g. schema v3's `reviewer_source` marked "Optional" while `review-judgment/SKILL.md`'s writer prose calls it a "required field") — were judged real but non-blocking (no rate corruption, narrow edge-case exposure) and filed as issue #842, not fixed in #768. One of these nits — the Optional-vs-required-field wording mismatch — was reconciled during #834's implementation: `review-judgment/SKILL.md`'s writer prose now reads "writer always emits this field (writer obligation); the reader/consumer treats an absent field as local ... for backward compat with pre-v3 entries (reader-semantics fallback)," reconciling the schema's "Optional" framing with the writer's obligation to always emit it. The remaining 13 findings in #842 are unaffected and still open.
- **Remote-keyed dedup uses key-equality, not the originally-specified latest-generation-wins (CM8, owner-decision pending).** `Get-ReviewCostRollup`'s review-dispositions dedup merges per-PR on the literal `stable_finding_key` value (768-D7); for a *remote* key (a GitHub comment ID), the same logical finding can legitimately carry a different key across re-review rounds, which key-equality dedup cannot detect or merge. This was judged a defensible v1 implementation of the spec's intent, not a code defect, and was routed to the maintainer as a plan-conformance question pending ratification rather than fixed silently.
- **True-containment gate change deferred, diagnosis-only fix shipped instead (see 817-D5, deferred).** The near-decoy window-bleed root cause — `Get-RealJudgeRulingsHeadMatches`'s fixed 400-char lookahead window, rather than a true containment bound at the next candidate's actual position — was not eliminated. `Get-JudgeRulingsDuplicateDiagnosis` (817-D1) only diagnoses, honestly and after the fact, why the M1 duplicate-head guard fired; no code change touched the underlying window mechanism itself, only the diagnosis/labeling layered on top of it. This was an explicit lite-scope owner decision (`lite-scope-bias-817`), not an oversight. The 817-D4 close-genuine-duplicates residual (T6) — resolving to `decoy-ambiguous` instead of the correct `head-corrupt` — is the same underlying imprecision the deferral leaves in place; it is accepted, fixture-pinned, and not eliminated. Mirror ordering (T7), also documented under 817-D4, is NOT part of this leftover imprecision: `decoy-ambiguous` is the objectively correct diagnosis for that single-real-head-plus-decoy placement, and T7 is a regression pin confirming correct behavior, not a residual.
- **The terminal-stage blind spot is closed (#854).** Before #854, code-review was the only stage in this ledger with no downstream observer: it structurally could not accrue escapes, so its escape rate read `0.00` and its relaxation-eligibility signal computed `ELIGIBLE` by construction — an artifact indistinguishable from a genuine zero, not a measurement. #854 ships `caught_stage: post-review-observer` (projection 4, escape_distance `4 - ordinal(catchable_phase)`) so external-reviewer catches the pipeline missed become real ledger escapes attributed to the code-review stage, closing the only surface where "no measurement" and "no defects" were structurally impossible to tell apart.
- **Two-arm certification model.** The code-review row now renders two independently-gated arms: the catch-side veto (any sustained critical- or high-severity code-review finding blocks eligibility, naming both counts and the triggering severities in the render) and a new escape-side miss estimate derived from post-review-observer entries, gated on `n>=5` co-observed PRs with a Chapman-or-sparse-estimator fallback below that threshold. Both arms must pass for `ELIGIBLE`; either arm alone downgrades the verdict to `NOT ASSESSABLE` or `WITHHELD` with a named reason — the escape side can only ever downgrade the catch side's verdict, never upgrade it. The catch-side veto's `high`-severity coverage is itself part of #854's own change (a judge-sustained M4 finding against this issue's shipped code): the veto and render previously checked only `critical`, so a sustained `high`-only window silently rendered `ELIGIBLE` despite AC4's requirement that critical/high catches both trigger the veto — it is not "unchanged from #762/#768." The Chapman estimator actually renders a third state beyond `sparse` (m<1) and a normal `estimate`: `unavailable`, which fires when the duplicate-match count `m` exceeds `min(n1, n2)` — the observed overlap between the two reviewers exceeds the smaller of their two observed catch populations, which makes the Lincoln-Petersen-style estimate meaningless (not merely imprecise), so the renderer reports it as unavailable rather than computing a nonsensical N-hat.
- **Fail-closed guard semantics: coverage means measurement, NaN-guard, reconciliation.** Three guards keep the escape-side arm honest and downgrade-only: (1) coverage counts a PR toward K when it has a judge-authored, cleanly-parsed review-dispositions record carrying an `external_sources_reconciled` coverage record — whether that list is empty or populated (M9: a head-present, clean parse with zero entries is itself a legal, measured coverage record; a review-cost-pass fix removed an earlier requirement that a PR also carry ≥1 external sustained finding resolved to a real identity, which wrongly zeroed out that legitimate zero-finding measurement); (2) the unique-catch-rate estimator explicitly rejects `$null`/`NaN` before any threshold comparison, because a naive divide-by-zero `NaN` satisfies neither `-lt` nor `-ge` and would otherwise read as `ELIGIBLE`; (3) the dispositions-recorded novel-external count is reconciled against the observer-block count counted directly from the fetched corpus (both sides counted catchable_phase-blind, since the dispositions record carries no catchable_phase field), and any mismatch — or a missing reconciliation input — forces the escape arm to `WITHHELD` rather than trusting either side alone. Both estimators' populations (n1's internal co-observed catches, and the Chapman supplement's n1/n2/m) are scoped to this same K population — PRs whose dispositions actually carry a reconciled coverage record — never to the wider whole-window PR population code-review already observes.
- **Two doctrine revisions, explicit and owner-approved.** `value-block-cost-dependency-854` intentionally revises #768's value/cost isolation for the code-review row only: the escape-side arm's coverage data is now hoisted from a single review-cost corpus fetch made once before the rollup call, with a `$corpusError` capture that re-throws into the existing cost-path try/catch so a fetch failure still degrades only the cost section and never corrupts the value report (both #768 test contracts re-pinned, not silently defeated). `coverage-authorship-854` closes the corpus-authorship forgery vector: the six comment-fetch query sites in `phase-containment-rolling-history-core.ps1` now select the comment author, so a coverage record must be judge-authored to count toward `MeasuredCoveragePRCount` — previously any comment matching the marker shape was trusted at face value, meaning a gate input anyone could type could unlock `ELIGIBLE`.

## Non-Goals

- Deciding to relax any review — that decision is gated on data this system produces and lives with the umbrella (#761).
- Enforcing emission (blocking a PR/issue on a missing ledger block) — the `-Mode enforce` parameter is accepted for forward compatibility but is currently unimplemented; the shipped behavior is warn-only regardless of the value supplied.
- Identity-precise `finding_key`-to-judge-ID correlation in the emission-check sweep — warnings are count-based (sustained N vs. blocks M); precise per-finding correlation is deferred.
- Extracting the duplicated Surface A/B marker-hunt state machine into a shared helper — tracked as #831, not re-solved here (772-D2, Known Limitations above).
- Bulk-backfilling the historic prose-only plan-stress-test corpus — explicitly rejected (811-D4): a mechanical block-derived backfill would be tautologically clean and mask real historic under-emission; only #794 was backfilled as the AC witness, and an agent-assisted honest bulk backfill would need to be filed as a separate #761 child if trend data is needed before the 90-day window ages the historic rows out.
- Deciding relaxation from the review-cost term — like the value ledger, #768's dismiss-rate/defense-kill-rate/defer term is presentation-only: it renders next to the existing escape/irreducible-catch rates but never feeds the relaxation-eligibility formula, which fires on value conditions alone (768's own worth-it decision scoped this as "Proceed-lite"; formula integration is deferred to #761 governance).
- Recovering cross-round logical identity for remote-keyed review-dispositions findings — dedup is latest-wins on the literal `stable_finding_key` value (768-D7), which is the best available fix for a key that legitimately varies across re-review rounds for a remote (GitHub comment ID) key, not a claim of perfect cross-round identity tracking (see Known Limitations, CM8).
- Live re-measurement of #863's plan-comment-cap split (AC2-M) inside #863's own PR — the plugin-cache release boundary means no honest post-change plan-comment artifact can exist until a version bump ships and is installed; deferred as a tracked close condition on umbrella #761 (owner decision `p-ac2-measurement-deferral`; see § Issue #863 above for the full record and the #761 comment link).
