# Design: Phase-Containment Escape-Rate Ledger

## Summary

Umbrella issue #761 asks a measurement question the adversarial review pipeline could not previously answer: for each review stage (design-challenge, plan-stress-test, code-review), what fraction of sustained findings escaped from an earlier phase that could have caught them, and what fraction were genuinely irreducible — catchable only at that stage? The maintainer's stated position is that no review is relaxed on a cost argument alone; a stage earns relaxation only when data shows its irreducible-catch rate has trended to ~0 over a rolling window with enough samples to trust.

The phase-containment ledger is the instrument that produces that data. Each sustained adversarial finding is annotated with `introduced_phase` (where the flaw originated), `catchable_phase` (the earliest phase it could reasonably have been caught), and `caught_stage` (which review actually caught it). The system was built across four issues:

- **#762** (sub-1) shipped the per-finding annotation schema, a hand-rolled parser/validator, a rolling-history walker with rollup, and a maintainer-facing report CLI.
- **#763** (sub-2) shipped the first escape-driven upstream fix: a four-quadrant grounding discipline in `design-exploration` that catches the class of design-phase miss the #760 seed entries recorded.
- **#772** shipped, closing all six findings PR #770's review deferred against the sub-1 reader: a capped (K=5) markerless-comment-page hunt with unbounded post-find pagination resume (772-D2), REST-fallback `createdAt` threading (restoring latest-annotation-wins dedup under REST), `finding_key` format validation (schema `pattern` + validator Rule 12, case-sensitive), total `Truncated`/`InvalidEntryCount` telemetry across every return shape of both public fetch functions (772-D1), and pair-matched open/close marker-block parsing so an unclosed block no longer corrupts the following block (772-D3). A post-implementation code-review pass (commit `6626f2f`) sustained four more findings — M1/M2 closed two additional silent-truncation paths (GraphQL post-marker-find pagination failures, and a REST list-level `gh` call failure that previously had no `else` branch at all), M4 folded D6's parser-level block skips into `InvalidEntryCount`, and M8 replaced a culture-hardcoded test literal. One deferred-significant maintainability finding (Surface A/B marker-hunt logic duplication) was filed as issue #831, not fixed in #772.
- **#782** closed the gap between "emission is specified" and "emission happens" for the design-challenge and code-review surfaces: a warn-only sweep that detects sustained findings missing their paired ledger blocks, plus the first organic backfill (16 entries recovered from PRs #775/#778/#781). It left the plan-stress-test surface's emission check silently false-clean (`sustained=0 blocks=0` on every plan, compliant or not) because it was written against a `judge-rulings` machine-block contract the plan-authoring writer never actually emitted.
- **#811** (this branch) fixed the plan-stress-test surface's false-clean: Issue-Planner now emits a machine-readable `judge-rulings` block alongside the human prose, the reader gains an honest fallback and fail-loud-on-duplicate-heads behavior, and issue #794 was backfilled as the acceptance-criteria witness (`sustained=11 blocks=11`, verified live).

This document describes the current shipped system. It does not cover relaxation-criterion governance (deferred to the umbrella, gated on this data).

## Design Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | Annotation home | One uniform `<!-- phase-containment-{ID} -->` block per finding, written by whichever stage caught it onto the surface it already posts to | The #762 design challenge found the plan-stress-test surface has no parseable block and `judge-rulings` is parsed by a bespoke line-scanner, not YAML — in-place extension of `finding_dispositions`/`judge-rulings` could not hold uniformly across all three surfaces |
| D2 | Escape-distance arithmetic | `escape_distance = projection(caught_stage) − ordinal(catchable_phase)`, validated by recomputation and rejected (not clamped) if the stored value differs, or if `introduced_phase > catchable_phase`, or if `catchable_phase` exceeds the caught stage's projection | A silently-clamped or unchecked distance would corrupt the escape/irreducible split the whole metric depends on |
| D3 | Fail-loud over silent-zero | Any unparseable, ambiguous, or unknown-vocabulary content under a recognized marker head is `could-not-verify`, never silently treated as zero findings or a clean gap | A metric whose failure mode is silent zero would misreport a stage as clean when it was actually unmeasured — the exact false-clean scenario S2 in #772/#782 exists to prevent |
| D4 | Undersample guard granularity | Per-stage `insufficient_data` (n<5) withholds that stage's relaxation-eligibility signal independently — not a single global window-count guard | A global guard would let a near-empty stage (e.g. plan-stress-test at n=1) read as computable simply because other stages had volume |
| D5 | `finding_key` surface discrimination | Uniform `{surface}:{stable_finding_key}` format; `Get-EmissionGap` requires a block's `finding_key` to be prefixed for the surface being checked before counting it toward that surface's block count | design-challenge and plan-stress-test marker heads can legitimately co-occur on the same issue body; without the prefix check, a block emitted for one surface could silently satisfy another surface's gap count |
| D6 | Warn-only, never-blocking posture | The emission-check sweep only ever posts an advisory comment; enforce mode is defined in the `-Mode` parameter but explicitly unimplemented (`warn` behavior runs regardless of the value supplied) | Mirrors the frame-credit-ledger warn→enforce staging precedent — surface the gap, never gate a review, PR, or issue update on it |
| D7 | Why emission enforcement needed a separate sweep | #762 shipped emission as a prose post-step in three skills (`design-exploration`, `plan-authoring`, `review-judgment`); every real review after PR #770 merged — #775, #778, #781, sustaining 13+ findings between them — emitted zero paired `phase-containment` blocks | An unenforced terminal instruction is reliably skipped; #782 built a reader-side reconciliation sweep as the durable guarantee rather than trusting a stronger prose reminder to succeed where the first one failed |
| D8 | Marker-vocabulary gating (not bare substring match) | A comment only counts as an authoritative judge-rulings/finding_dispositions surface when its marker head is followed, within a bounded lookahead window, by recognizable field-vocabulary tokens (`disposition`, `judge_ruling`, `verdict`, `finding_key`, `finding_id`, `schema_version`) at a real YAML key position (line-start, flow-mapping, or dash-space list-item) | Early iterations mistook ordinary prose describing the marker convention, or dash-space list-item findings, for real marker content — both a false-could-not-verify and a false-clean failure mode had to be closed |
| D9 | Backfill-append primitive is read-modify-write, never upsert | `Add-CommentBlocks` does `gh api` GET → verify marker → concatenate → PATCH → positive-proof re-verify; it explicitly does not use `Find-OrUpsertComment`, whose PATCH path replaces the body verbatim | The judge-rulings YAML that Code-Conductor's credit harvest reads lives in the same comment the backfill blocks are appended to — an upsert-style replace would destroy it |
| D10 | Post-write verification is positive-proof, not byte-prefix | Verification confirms (a) the original marker survived and (b) every appended block is present, parseable, and content-identical in the re-fetched body — not an exact `StartsWith` byte comparison | Live backfill against #775/#778/#781 showed GitHub's API benignly normalizes whitespace on write/read, which broke ordinal-prefix comparison and produced false negatives that would have trained the caller to ignore a real fail-loud signal |
| 811-D1 | Plan-stress-test writer emits a machine `judge-rulings` block, not a prose parser | `skills/plan-authoring/SKILL.md` § Judge-rulings machine block: at Post-Judge Reconciliation, Issue-Planner appends a `<!-- judge-rulings` block (one entry per merged `finding_id`, binary `sustained \| defense-sustained` projection, atomic single write) at the end of the `<!-- plan-issue-{ID} -->` comment, alongside the unchanged human-facing `**Plan Stress-Test**` prose | #782's design already specified that the plan-stress-test surface persists a `judge-rulings` block and the reader's parser branch was written against that contract — the actual defect was a writer deviation (the block was never emitted), not a missing parser. Live #794 prose also proved not machine-countable (aggregated bullets, out-of-enum vocabulary), which foreclosed a prose-parser fix |
| 811-D2 | Reader honest fallback, differentiated by cause | `Test-EmissionMarkerPresent`, plan-stress-test surface only: a body carrying `<!-- plan-issue-` and a line-start `**Plan Stress-Test**` heading but no judge-rulings head reports could-not-verify instead of falling through to `sustained=0` chatter-skip. `Format-EmissionGapLine` renders two distinct could-not-verify messages — `head-missing` (blocks may already exist; a one-line backfill candidate) vs. `head-corrupt` (a machine head exists but failed to parse) — and renders a negative gap (more blocks than sustained) as a loud excess, never as `clean` | A prose-only plan (every plan persisted before #811) previously read as ordinary chatter and silently contributed `0/0`, indistinguishable from an issue with no stress-test history at all — exactly the false-clean scenario S2/S3 this issue was filed to close |
| 811-D3 | Fail-loud on duplicate judge-rulings heads (not latest-wins) | When two or more real (vocab-gated) judge-rulings heads exist in one comment body, the reader returns `could-not-verify` rather than picking one | Owner decision during the plan's own adversarial stress-test: latest-wins fights the existing ambiguity-walk / attributed-vs-bare head precedence logic, and the writer's replace-own-block rule already guarantees a single head on the normal persist path — a duplicate signals a genuine anomaly (e.g. a double-run backfill) that should surface loud, not resolve quietly |
| 811-D4 | Historic corpus: no bulk backfill, #794 only | Only issue #794 was backfilled (17 entries via the sibling `Add-JudgeRulingsBlock` primitive) as the acceptance-criteria witness; the other ~34 of ~39 closed plan issues stay prose-only and render the differentiated could-not-verify variant until they age out of the 90-day corpus window | Honest backfill of historic plans requires per-issue prose re-reading; a mechanical block-derived bulk backfill would be tautologically clean (sustained ≡ blocks by construction) and would mask real historic under-emission — owner decision `historic-corpus-disposition-811` |
| 772-D1 | Truncation/invalid-entry telemetry needed a real return channel | Reshaped the private fetch helpers (`Get-SurfaceACorpusGraphQL`/`Get-SurfaceBCorpusGraphQL`/`Get-PhaseContainmentCorpusRest`) and `Invoke-PhaseContainmentCommentScan` from bare arrays/`$null` into carrier objects (`{Tuples/Entries; Truncated; InvalidEntryCount; IsError}`, explicit `IsError` replacing the old `$null`-means-REST-fallback sentinel), then threaded `Truncated`/`InvalidEntryCount` through all 8 `Get-PhaseContainmentHistory` returns and all 7 `Get-PhaseContainmentCommentCorpus` returns | A bare-array/`$null` return had no channel to carry a truncation signal out of the private fetch layer; the plan's own stress-test flagged this as the load-bearing propagation gap (challenge P2, sustained) that every other truncation fix in the plan depended on |
| 772-D2 | Marker hunt is capped (K=5), not unbounded — but post-find collection is not | `Get-SurfaceACorpusGraphQL`/`Get-SurfaceBCorpusGraphQL` hunt up to 5 additional comment pages for a phase marker before giving up (GraphQL only; REST returns all comments in one call); once a marker is found, pagination resumes UNBOUNDED to collect all remaining comment pages before recording the tuple | An unbounded hunt on every markerless thread would regress the primary read path's performance (F11's no-perf-regression clause); the plan-stress-test caught that capping the hunt without also uncapping post-find collection would silently drop a `phase-containment-{N}` block sitting past page 5 on a thread where the marker itself was found early (challenge P6, sustained) |
| 772-D3 | Parser-level block skip must warn, not just drop | `Get-PhaseContainmentBlock` pair-matches open/close marker tags (an unmatched open tag followed by a later open tag drops the earlier block as malformed) and now emits `Write-Warning` plus an optional `-SkippedCount` `[ref]` out-param that `Invoke-PhaseContainmentCommentScan` folds into `InvalidEntryCount` | The design-challenge caught that D6's original fix — silently dropping the unclosed block — was itself a new silent-degradation path if left unflagged (challenge P10, sustained), inconsistent with the ledger's own "no path lacks a fix, flag, or fail-closed outcome" bar |

## Metric Definitions and Scope Boundary

The ledger measures, per review stage, over a rolling window:

- **Escape rate** — the fraction of findings whose `catchable_phase` denominator stage leaked (caught later than that stage).
- **Irreducible-catch rate** — the fraction caught exactly at their `catchable_phase` stage, genuinely requiring that stage's information.
- **Relaxation-eligibility signal** — emitted only when irreducible rate trends to ~0 **and** the stage's window sample size is `n ≥ 5` **and** no critical-severity finding was caught in the window. Below n=5 the signal is withheld and the stage reports `INSUFFICIENT DATA` rather than a false confident rate.

The system is explicitly a measurement instrument, not an enforcement or relaxation mechanism:

- It never blocks a PR, issue, or review on missing or gapped ledger data (D6).
- It does not itself decide to relax any review; that decision is deferred to the umbrella once the data supports it.
- The emission-check sweep does not verify comment authorship — any comment matching the recognized marker shapes is trusted at face value; this is an accepted residual-risk tier for a warn-only maintainer advisory, not a compliance guarantee.

## Current Status and Known Limitations

Corpus totals have grown substantially since this document was first written and now reflect accumulated activity across many issues and PRs rather than the small seed+organic counts originally recorded here; see `phase-containment-report.ps1`'s live output for current per-stage `n`, escape rate, and irreducible rate. This section focuses on what #811 changed for the plan-stress-test surface specifically.

Before #811, the plan-stress-test surface's emission check (`phase-containment-emission-check.ps1`) was silently false-clean: it reported `sustained=0 blocks=0` on every plan regardless of how many sustained findings and phase-containment blocks actually existed, because it parsed the surface with the code-review `judge-rulings` YAML parser against a plan comment that only ever carried human prose bullets. Issue #794 was the live reproduction (11 sustained findings, 11 real blocks, reported as `0/0`) and is now the shipped fix's acceptance-criteria witness — after the plan-authoring writer change and the #811-D4 backfill, `phase-containment-emission-check.ps1 -Issue 794` reports `plan-stress-test #794: clean -- sustained=11 blocks=11` (verified live). The plan-stress-test surface's trend baseline (relaxation-eligibility signal, escape rate) starts accumulating organic, honestly-counted data from #811's ship date forward; historic prose-only plans are not retroactively counted (811-D4) and age out of the rolling window naturally.

Known limitations:

- **Surface A/B marker-hunt logic is duplicated between the two GraphQL fetch helpers.** The K=5 capped hunt + unbounded-resume-after-find state machine (772-D2) was added near-verbatim (~65-90 lines) to both `Get-SurfaceACorpusGraphQL` and `Get-SurfaceBCorpusGraphQL` in `phase-containment-rolling-history-core.ps1`, extending a pre-existing Surface A/B mirroring pattern already in the file; Surface B's Pester coverage of the five hunt dispositions (marker-on-page-1, marker-found-mid-hunt, cap-exhausted, mid-hunt-timeout, natural-exhaustion) is also thinner than Surface A's. The #772 post-fix code-review judge ruled this non-blocking for merge (defense-sustained, categorized structural) since no #772 acceptance criterion covered extraction; it was filed as issue #831 as a maintainability follow-up, not a correctness gap.
- **The emission-check's single-target mode has no dry-run flag.** `phase-containment-emission-check.ps1 -Pr N` or `-Issue N` always posts its advisory report via `Find-OrUpsertComment` — there is no read-only mode that computes the gap without writing a comment. This was discovered during #782's CE Gate pass. It is a usability gap worth fixing, not a defect: the report is idempotent (find-or-upsert on a fixed marker) and warn-only, so repeated runs do not compound, but a maintainer who wants to preview the gap without touching the PR/issue currently cannot.
- **Historic plan-stress-test corpus is prose-only and will read as could-not-verify.** ~34 of ~39 closed plan issues predate #811's writer change and carry no machine `judge-rulings` block. `Format-EmissionGapLine` renders these as the differentiated `head-missing` could-not-verify variant (naming the existing block count as a one-line backfill candidate) rather than a false `clean`, but no bulk backfill was performed (811-D4, owner decision) — only #794 was backfilled as the AC witness.

## Non-Goals

- Deciding to relax any review — that decision is gated on data this system produces and lives with the umbrella (#761).
- Enforcing emission (blocking a PR/issue on a missing ledger block) — the `-Mode enforce` parameter is accepted for forward compatibility but is currently unimplemented; the shipped behavior is warn-only regardless of the value supplied.
- Identity-precise `finding_key`-to-judge-ID correlation in the emission-check sweep — warnings are count-based (sustained N vs. blocks M); precise per-finding correlation is deferred.
- Extracting the duplicated Surface A/B marker-hunt state machine into a shared helper — tracked as #831, not re-solved here (772-D2, Known Limitations above).
- Bulk-backfilling the historic prose-only plan-stress-test corpus — explicitly rejected (811-D4): a mechanical block-derived backfill would be tautologically clean and mask real historic under-emission; only #794 was backfilled as the AC witness, and an agent-assisted honest bulk backfill would need to be filed as a separate #761 child if trend data is needed before the 90-day window ages the historic rows out.
