# Cross-tool Handoff Markers

Durable GitHub comment markers used by Agent Orchestra for phase handoffs, smart resume, engagement records, and credit ledger. Markers are HTML comments written to GitHub issue or PR threads; because they live on the issue, work resumes across sessions without losing context.

Row-level survival and fallback semantics: [../SKILL.md](../SKILL.md) (session-memory-contract).  
Persistence rationale: [../../../Documents/Design/session-memory-contract.md](../../../Documents/Design/session-memory-contract.md).

## Writing about markers safely

**The hazard**: every marker family in this document is detected by a raw-text scan (`-match`, `[regex]::IsMatch`, `.Contains`) against a comment or PR body — not a semantic parse. Markdown code-span backticks do NOT neutralize these scans: a complete marker literal such as the live delimited form of `judge-rulings pr=5` still matches even inside backticks, because the regex scans raw text and backticks are not stripped before matching. Any prose that quotes a marker literal verbatim — explaining the convention, giving a worked example, or describing what a comment "carries" — is therefore live to the same reader that scans real comments, and can produce a false positive (a prose mention miscounted as a real block or head) or, via `Find-OrUpsertComment`'s `-like` substring matcher, cause an entirely wrong comment to be selected as the target for a write.

**Families affected**: every marker family whose reader does a raw-text scan is affected — this covers every family in the Active Marker Families list above, including `pipeline-metrics`, `cost-pattern-data`, `phase-containment`, `phase-containment-ledger`, `frame-spine`, `frame-slice`, `credit-input`, `engagement-record`, `judge-rulings`, `plan-issue`, `design-phase-complete`, and `experience-owner-complete`. Assume the same hazard applies to any new marker family introduced later unless its reader is proven to do a genuine semantic parse (e.g., a GraphQL comment-id lookup) rather than a raw-text scan.

**The remedy**: use `Format-InertMarkerLabel` (`.github/scripts/phase-containment-emission-check.ps1:148-152`) to render a marker mention in prose. It strips the `<!--`/`-->` HTML-comment delimiters before backtick-wrapping, which genuinely breaks the regex match because the delimiters the pattern anchors on are no longer present in the text at all.

**Worked example**: rendering the live `judge-rulings pr=5` marker for prose reads as `` `judge-rulings pr=5` `` — never with the `<!--`/`-->` delimiters left intact inside the backticks. The same discipline applies to every marker family listed above: write `` `phase-containment-ledger-ref: 123` ``, not the delimited form of that same literal.

This rule was first written for the plan-surface `judge-rulings` block (`skills/plan-authoring/SKILL.md` § Judge-rulings machine block, writer rule 5), which now cross-references this section rather than duplicating it.
