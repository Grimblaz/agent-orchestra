# Cross-tool Handoff Markers

Durable GitHub comment markers used by Agent Orchestra for phase handoffs, smart resume, engagement records, and credit ledger. Markers are HTML comments written to GitHub issue or PR threads; because they live on the issue, work resumes across sessions without losing context.

Row-level survival and fallback semantics: [../SKILL.md](../SKILL.md) (session-memory-contract).  
Persistence rationale: [../../../Documents/Design/session-memory-contract.md](../../../Documents/Design/session-memory-contract.md).

## Active Marker Families

- `<!-- experience-owner-complete-{ID} -->` — upstream framing complete
- `<!-- design-phase-complete-{ID} -->` — technical design complete
- `<!-- engagement-record-experience-{ID} -->` — durable engagement audit for /experience phase: load-bearing decisions, audit rationale, articulation text persisted alongside the experience-owner-complete marker for cross-session decision memory (SMC-20)
- `<!-- engagement-record-design-{ID} -->` — durable engagement audit for /design phase: load-bearing decisions persisted alongside the design-phase-complete marker; consumed by solution-authoring's same-decision-resume rule on phase re-entry (SMC-20)
- `<!-- engagement-record-plan-{ID} -->` — durable engagement audit for /plan phase: load-bearing decisions persisted alongside the plan-issue marker; consumed by solution-authoring's same-decision-resume rule on phase re-entry (SMC-20)
- `<!-- engagement-record-orchestration-{ID} -->` — durable engagement audit for orchestration touchpoint (`scope-classification`): persisted as an issue comment when scope-classification resolves; payload Markdown mirror co-located in the comment; consumed by solution-authoring's same-decision-resume rule on Code-Conductor re-entry (SMC-20)
- `<!-- engagement-record-review-{PR} -->` — durable engagement audit for /orchestra:review and /orchestra:review-judge phases: load-bearing review-finding dispositions persisted as a PR comment after the post-judge disposition gate completes; consumed by same-decision-resume on re-review of the same PR (SMC-20, SMC-23, schema_version 4)
- `<!-- review-dispositions-{PR} -->` — per-finding disposition record for PR code-review verdicts; one entry per judge-sustained finding carrying stable_finding_key, pass, disposition (incorporate|dismiss|escalate), classification, and disposition_rationale (SMC-23)
- `<!-- design-issue-{ID} -->` — durable design snapshot handoff used for D9 pause/resume and full-pipeline smart resume
- `<!-- plan-issue-{ID} -->` — approved plan persisted
- `<!-- frame-credit-ledger-{PR} -->` — warn-only frame credit-ledger comment posted by the pre-PR hook (sub-issue #429 of frame umbrella #425); idempotently upserted on every PR after `gh pr create`
- `<!-- review-judge-produced-{PR} -->` — sentinel written by the judge (both Copilot and Claude) immediately after the ruling finalizes, before pipeline-metrics persistence; the warn-only hook detects this to synthesize a `not-persisted` review credit when the PR body carries no review credit yet (SMC-16)
- `<!-- credit-input-{port}-{ID} -->` — deferred-emission marker written by pipeline-entry agents (Experience-Owner, Solution-Designer, Issue-Planner) immediately after their completion marker; payload is a `yaml` fenced block carrying `{ port, adapter, evidence }`; harvested by Code-Conductor at PR-creation time to emit the corresponding credit row (SMC-17)
- `<!-- proposed-followups-{PR|ISSUE} -->` — headless-queue fallback for the maintainer-approval filing gate (issue #837): posted as a PR or issue comment when no interactive surface is available, carrying proposed follow-up issues as a fenced YAML payload; transitions through `proposed` → `claimed` → `consumed` states. The `followup-` durable drop/modify records reuse the existing SMC-20 (issue-keyed) and SMC-23 (`engagement-record-review-{PR}`, PR-keyed) markers additively — this marker covers only the headless queue payload itself, not the drop/modify records (SMC-24)
- `<!-- frame-slices-{ID} -->` — sibling comment (issue #863) carrying every `<!-- frame-slice ... -->` block for the plan's implementation steps, plus a `frame-slices-generated-at` stamp that must equal the spine's `generated_at` value. **Writer**: Issue-Planner, at plan persist and re-persist (`skills/plan-authoring/SKILL.md` § Spine and Slice Discipline). First persist creates the sibling and records its comment id back onto the plan comment's `frame-spine` block as `slice_comment_id` (863-D3); re-persist reuses the existing sibling (found via its identity marker or the existing pointer) rather than creating a second one. **Reader**: Code-Conductor and Spine-Runner, via the `frame-spine-lookup` two-fetch-and-concatenate shim — the shim fetches the plan comment plus this sibling by the `slice_comment_id` Dispatch Input and concatenates the bodies before invoking `-Op Lookup`; the shim never parses the spine itself, and `frame-validate-core.ps1`'s structural checks (port-to-slice coverage) likewise expect the concatenated form when run manually against a split plan (`frame-validate -Mode plan` is manual-only; see `Documents/Design/phase-containment-ledger.md` for the exact error a bare, unconcatenated plan comment produces). `Invoke-FSCSpineLookupCli` cross-checks the sibling's `frame-slices-generated-at` stamp against the spine's `generated_at` and fails loud (`stale-spine`) on divergence, on pointer-present/marker-absent, and on a duplicate slice id across the concatenated corpus. **Survival semantics**: `durable` (GitHub issue comment on the subject issue); part of the plan-comment family the SMC-01 row governs, split out for reader-specific fetch scoping rather than a new persistence mechanism (no-new-mechanism rule). Legacy (pre-#863) plans carry no `slice_comment_id` and have no sibling — the lookup shim's single-fetch path stays the behavior for that population.
- `<!-- phase-containment-ledger-{ID} -->` — sibling comment (issue #863) carrying every `<!-- phase-containment-{ID} -->` block for the plan's sustained plan-stress-test findings, plus the plan-surface `<!-- judge-rulings ... -->` machine block — the two families co-move together (863-D4) so Fix A's co-location gate (`phase-containment-emission-check-core.ps1`, issue #782) stays satisfied: a `judge-rulings` head only authorizes counting the `phase-containment` blocks that share its own comment body. **Writer**: Issue-Planner, at Post-Judge Reconciliation (`skills/plan-authoring/SKILL.md` § Phase-containment emission, § Judge-rulings machine block). First persist creates the sibling — its body opens with the identity marker — and records its comment id back onto the plan comment as a standalone `<!-- phase-containment-ledger-ref: {comment_id} -->` marker placed immediately after the `<!-- plan-issue-{ID} -->` marker (863-D11); re-persist replaces the prior `judge-rulings` block on the sibling in place (never appends a second one) and appends any new `phase-containment` blocks. **Reader**: the phase-containment emission-check subsystem (`phase-containment-emission-check.ps1`, `Get-EmissionGap`, `Get-SustainedFindingCount`) and the rolling-history walker (`phase-containment-rolling-history-core.ps1`) — both fetch every comment on the issue via `gh api`/GraphQL regardless of which comment carries the blocks, so the split does not change either reader's fetch cost (see `Documents/Design/phase-containment-ledger.md` for why the once-projected 5.6x reader-waste figure was retired as unactionable). **Survival semantics**: `durable` (GitHub issue comment on the subject issue); part of the plan-comment family the SMC-01 row governs. Legacy (pre-#863) plans carry no `phase-containment-ledger-ref` pointer and no sibling — the 811-D1 fallback in `Test-EmissionMarkerPresent` continues to read the plan comment's own prose/`judge-rulings` head directly for that population. **Write-only, navigation-only (863 M2/M17 fix)**: `phase-containment-ledger-ref` exists to help a human or agent find the sibling; no reader may treat the pointer's mere presence as evidence that a real head or block exists anywhere — `Get-EmissionGap`'s 811-D1 suppression gate binds on the `phase-containment-ledger-{Id}` identity marker actually being present on the qualifying body, never on the pointer alone, and any future "optimization" that trusts the pointer without also checking the marker match reopens the exact false-suppression vector the M2 fix closed.
